# Release notes template

The `publish-releases` action already aggregates per-package CHANGELOG entries into the body of every date-tagged release. This template only describes how to **wrap** that body — and only for releases that include at least one minor/major bump. Patch-only releases stay flat.

## Patch-only release (every package bumped is a patch)

Leave the auto-generated body as-is. No headline, no lead, no footer. The per-package `### @subsquid/...` sections plus the Contributors footer are sufficient.

## Minor / major release (at least one minor or major bump)

Wrap with a headline, a lead paragraph, and a compare link. Keep the auto-aggregated body verbatim in between.

```markdown
## <Headline>

<1-3 sentence lead — what changed for the user, not the implementation.>

### @subsquid/<pkg-A> <ver>
... (auto-generated by publish-releases, untouched)

### @subsquid/<pkg-B> <ver>
...

---

### Contributors

@user1, @user2

**Full Changelog**: https://github.com/subsquid/squid-sdk/compare/<PREV-DATE-TAG>...<NEW-DATE-TAG>
```

## Style rules

- **No emoji** — neither in the title nor in section headers or bullets.
- **No version in the title** — the tag (`YYYY-MM-DD`) is rendered separately by GitHub.
- **No package list in the title** — the headline names the *thing that changed*, not the packages.
- **Lead with user-visible impact**, not internal mechanism. "EVM processor retries on transient archive errors" beats "Wrap fetcher in retry middleware".
- **Don't restate every package**. The per-package sections are already in the body. The headline is editorial framing for the largest user-facing change. If two changes are equally important, pick the one users will hit first.
- **Don't rewrite the auto-generated `### @subsquid/<pkg> <ver>` sections.** They come straight from each package's CHANGELOG.md and are the source of truth.
- **Resolve the previous tag** with `gh api repos/subsquid/squid-sdk/releases --jq '.[1].tag_name'` (index 0 is the release you just cut). Don't construct it from a date — same-day re-runs use `.2`, `.3` suffixes and the simple `YYYY-MM-DD - 1 day` math gets it wrong.
- **Skip the wrap-around if you can't write a meaningful headline.** A patch release that fixes three unrelated bugs across three packages doesn't have one — leave it bare.

## Headline examples

Good:
- `## EVM processor: faster reorg recovery`
- `## TypeORM Store: native Postgres LISTEN for hot-path updates`
- `## Solana RPC: streamed block decoding`

Bad:
- `## v2026-04-15` (no version in title)
- `## 🚀 Big release!` (no emoji, no marketing copy)
- `## Bug fixes and improvements` (says nothing)
- `## evm-processor 1.29.0, typeorm-store 1.8.0, ...` (don't list packages)
